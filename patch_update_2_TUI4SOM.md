# Патч обновления №2 для TUIopenMath / SysOpenMath

## Назначение патча

Данный патч является продолжением предыдущего этапа развития системы и направлен на приближение проекта к модели управляемой вычислительной платформы, аналогичной по духу SysOpenMath.

Основная цель патча состоит в следующем:

1. перевести систему на работу со **штатными профилями**, поддерживаемыми разработчиком;
2. сделать проверки компонентов **профильно-зависимыми**;
3. устранить ложные ошибки диагностики для компонентов, не входящих в выбранный профиль;
4. улучшить запуск JupyterLab;
5. оформить визуальную часть запуска через splash-экран.

При подготовке данного патча учитывается ранее зафиксированная архитектурная линия: JSON-профили рассматриваются как **внутренний конфигурационный формат системы**, а не как пользовательский интерфейс свободного конструирования среды.

---

## Основные изменения

### 1. Переход к модели штатных профилей

Система переводится на использование заранее определённых профилей, поддерживаемых разработчиком.

Пользователь:
- не формирует вычислительную среду вручную;
- не конструирует профиль произвольным образом;
- не управляет внутренней структурой конфигурации.

Пользователь только:
- выбирает один из доступных профилей;
- запускает подготовку среды;
- выполняет проверку состояния компонентов;
- использует готовую вычислительную среду.

Таким образом, профиль рассматривается как **штатный сценарий развёртывания**, а не как объект свободного пользовательского редактирования.

---

### 2. Единое действие полной подготовки выбранного профиля

Ранее логика полной подготовки была частично завязана на конкретные профили, такие как:

- `base`
- `math`
- `ml`

В рамках данного патча вводится единое действие:

```text
Подготовить выбранный профиль полностью
```

Это означает, что система больше не должна опираться на жёстко зашитые ветвления вида:

- `if profile == base`
- `if profile == math`
- `if profile == ml`

Вместо этого используется единый сценарий подготовки для текущего выбранного профиля.

---

### 3. Профильно-зависимая проверка компонентов

Одним из главных изменений патча является переделка механизма диагностики.

Теперь система должна проверять **только те компоненты, которые реально входят в выбранный профиль**.

#### Проверка секций профиля

Для каждого профиля отдельно учитываются:

- `python`
- `julia`
- `system`
- `kernels`

Это означает:

#### Если в профиле указано:

```json
"python": ["sympy", "matplotlib", "jupyterlab"]
```

то система проверяет только эти Python-пакеты.

#### Если в профиле указано:

```json
"julia": ["IJulia", "Symbolics"]
```

то система проверяет только эти Julia-пакеты.

#### Если в профиле указано:

```json
"system": []
```

то система не должна выдавать ошибки по системным компонентам, отсутствующим в системе, если они не требуются данному профилю.

#### Если в профиле указано:

```json
"kernels": ["python", "julia", "bash"]
```

то система проверяет только эти kernels.

---

### 4. Замена ложных ошибок на логическое пропускание проверки

До патча диагностика могла показывать сообщения вида:

```text
[FAIL] scilab command
[FAIL] scilab kernel
```

даже в тех случаях, когда Scilab вообще не входил в выбранный профиль.

После патча такая логика должна быть устранена.

Вместо ложных ошибок система должна:

- либо полностью пропускать проверку не относящихся к профилю компонентов;
- либо явно указывать, что данный компонент не требуется выбранному профилю.

Пример корректного поведения:

```text
[SKIP] no system dependencies required
[SKIP] scilab kernel not required for profile symbolic
```

---

### 5. Введение проверки готовности профиля

Патч усиливает идею проверки не отдельных компонентов как таковых, а готовности всей среды выбранного профиля.

Для этого вводится понятие:

- `NOT READY`
- `PARTIALLY READY`
- `READY`

Профиль считается готовым к работе только в том случае, если:

- существует его Python-окружение;
- доступны обязательные зависимости Python;
- доступны обязательные зависимости Julia;
- зарегистрированы необходимые kernels;
- выполнены все обязательные условия, определённые содержимым профиля.

Пример итогового сообщения:

```text
[OK] profile readiness: READY
```

---

### 6. Улучшение запуска JupyterLab

Ранее запуск JupyterLab сопровождался попыткой открытия временного HTML-файла, что приводило к нестабильному поведению и ошибкам вида:

```text
Файл не найден
.../jpserver-xxxxx-open.html
```

В рамках патча логика запуска JupyterLab должна быть переведена на более устойчивую модель.

#### Новая модель запуска

- JupyterLab запускается в фоне;
- браузер не открывается автоматически;
- сервер запускается в режиме:

```bash
jupyter lab --no-browser
```

- система извлекает URL доступа с токеном из лога запуска;
- прямая ссылка выводится пользователю в журнал.

Пример ожидаемого поведения:

```text
[INFO] JupyterLab started in background
[INFO] Access URL: http://127.0.0.1:8888/lab?token=...
```

Такой подход:
- надёжнее;
- не зависит от временного HTML-файла;
- лучше соответствует TUI-модели работы.

---

### 7. Splash-экран при запуске

В рамках патча вводится splash-экран с логотипом системы.

#### Назначение splash-экрана

- визуально оформить запуск платформы;
- показать фирменный логотип;
- отделить этап старта от основного рабочего интерфейса.

#### Поведение splash-экрана

- splash отображается до нажатия пользователем любой клавиши;
- при недостаточном размере окна должен использоваться fallback на упрощённый текстовый вариант;
- основной интерфейс отображается только после завершения splash-экрана.

Это позволяет:
- не загромождать рабочее окно логотипом;
- сохранить аккуратную структуру интерфейса;
- улучшить первое впечатление от запуска программы.

---

### 8. Повышение согласованности интерфейса с новой архитектурой

Данный патч также усиливает соответствие интерфейса новой модели системы.

Интерфейс теперь должен транслировать идею:

- не конструктор;
- не свободное пользовательское конфигурирование;
- а **управляемая платформа с готовыми режимами работы**.

Это означает, что основной сценарий использования системы становится следующим:

1. выбрать профиль;
2. подготовить выбранный профиль полностью;
3. проверить компоненты;
4. работать в готовой среде.

---

## Архитектурный эффект патча

После внедрения данного патча система становится:

- более управляемой;
- более предсказуемой;
- ближе к модели штатных профилей;
- менее шумной с точки зрения диагностики;
- более логичной в плане пользовательского сценария;
- ближе к архитектурной задумке SysOpenMath.

Проверка компонентов теперь привязана не к “всей системе вообще”, а к **содержимому конкретного профиля**, что делает систему более зрелой и более соответствующей декларативной модели профилей.

---

## Практический смысл изменений

### До патча

- система могла проверять компоненты, не относящиеся к выбранному профилю;
- возникали ложные ошибки диагностики;
- запуск JupyterLab был завязан на временные HTML-файлы;
- большая визуальная графика пыталась отображаться внутри рабочего окна;
- полная подготовка ещё частично зависела от старых жёстких сценариев.

### После патча

- проверяются только компоненты, реально входящие в профиль;
- readiness определяется по обязательным зависимостям выбранного профиля;
- ложные ошибки по лишним компонентам устраняются;
- JupyterLab запускается устойчиво и отдаёт прямую ссылку в лог;
- splash-экран отделяет этап запуска от рабочего интерфейса;
- пользователь работает с готовыми штатными сценариями.

---

## Что необходимо проверить после применения патча

### 1. Проверка профиля `symbolic`

Следует убедиться, что для профиля:

```json
{
  "name": "symbolic",
  "python": ["sympy", "matplotlib", "jupyterlab"],
  "julia": ["IJulia", "Symbolics"],
  "system": [],
  "kernels": ["python", "julia", "bash"]
}
```

проверяются только:

- `sympy`
- `matplotlib`
- `jupyterlab`
- `IJulia`
- `Symbolics`
- `python kernel`
- `julia kernel`
- `bash kernel`

И не проверяются как обязательные:

- `octave`
- `scilab`
- `octave kernel`
- `scilab kernel`

---

### 2. Проверка readiness

После подготовки профиля необходимо убедиться, что система выводит итоговый статус готовности.

Для корректно подготовленного профиля ожидается:

```text
[OK] profile readiness: READY
```

---

### 3. Проверка запуска JupyterLab

Необходимо проверить, что:

1. JupyterLab запускается в фоне;
2. временный HTML-файл не используется как основной способ доступа;
3. в лог попадает прямая ссылка с токеном;
4. ссылка открывается в браузере вручную и ведёт в JupyterLab.

---

### 4. Проверка splash-экрана

Следует убедиться, что:

- splash отображается до нажатия клавиши;
- логотип корректно центрируется;
- при малом размере окна используется упрощённый вариант.

---

### 5. Проверка работы новых профилей

Нужно проверить, что действие:

```text
Подготовить выбранный профиль полностью
```

работает не только для старых профилей, но и для новых, например:

- `symbolic`
- `numerical`

---

## Ограничения

Следует учитывать, что данный патч по-прежнему является этапом развития системы, а не окончательной завершённой реализацией.

Полностью зрелая модель будет достигнута после внедрения:

- единого механизма статусов компонентов;
- полноценной логики:
  - `check`
  - `compare version`
  - `decide`
  - `install`
- разделения платформенных и профильных проверок;
- унифицированной версии smart-install для всех менеджеров компонентов.

---

## Следующий этап развития

Логическим продолжением данного патча является:

1. введение общей модели `InstallStatus` для всех компонентов;
2. реализация проверки версии для:
   - Julia
   - Jupyter
   - Octave
   - Scilab
3. развитие логики `SKIP / READY / BROKEN / MISMATCH`;
4. отделение глобальной платформенной диагностики от профильной диагностики;
5. дальнейшее упрощение интерфейса под сценарий штатной платформы.

## Итог

Данный патч переводит систему на более зрелую архитектурную модель, в которой:

- профили становятся штатными сценариями работы;
- проверка компонентов зависит от реального содержимого профиля;
- устраняются ложные диагностические ошибки;
- JupyterLab запускается устойчивым способом;
- интерфейс и UX становятся ближе к модели управляемой вычислительной платформы.

Таким образом, проект становится существенно ближе к исходной архитектурной задумке SysOpenMath.
